# Batch — Phase 7 followup: Suggested next prompts

A new feature for chat: three pill suggestions above the composer that each represent a possible next directive. Tap a pill → the composer fills with a detailed prompt body that the user can edit and send (or send as-is).

Suggestions are generated by a second LLM call that fires automatically after the main assistant response completes. Background, non-blocking. By the time the user finishes reading, suggestions are likely ready.

Re-read CLAUDE.md before starting. Disk-avoidance unaffected.

---

## Critical: tablet UX

Pills are tap targets. Each pill is ≥48px tall, comfortable to tap. Three pills horizontal on wide viewports; stack vertically on narrow ones.

The feature should feel ambient — present when useful, invisible when not. Don't fight the user.

---

## Required changes

### Part 1 — Schema: cache suggestions per assistant message

Suggestions are tied to a specific assistant message. When you switch branches via chevrons, the suggestion set should match whichever branch is now active. To avoid regenerating on every branch flip, cache.

Two storage options:

- **(a) DB column on Message.** `Message.suggestionsJson` — Json? — populated when suggestions complete, re-used on revisit.
- **(b) In-memory only.** Cache in client state; reload means regenerate.

**(a) is right.** Suggestions take 2-5 seconds to generate; persisting them avoids the user staring at skeletons every time they re-open a chat. Schema cost is one nullable Json column.

`prisma/schema.prisma`:

```prisma
model Message {
  // ... existing fields ...
  suggestionsJson Json?    // SuggestionRecord[] | null
}
```

Apply via `npx prisma db push`. Existing messages backfill with null.

### Part 4 — System prompt for the suggester

`src/lib/writerSuggestionsPrompt.ts`:

```ts
export const SUGGESTIONS_SYSTEM_PROMPT = `You are a story analyst suggesting where a collaborative fiction could go next.

You will be shown the conversation so far between a director (who gives brief directives) and a writer (who expands them into prose). Your job is to suggest three different directions the story could move next.

Output exactly three suggestions in this format:

[SUGGESTION 1]
LABEL: <3-8 word summary of the direction, written as a short phrase, no period>
PROMPT: <30-60 word detailed directive in the director's voice, written as a single instruction. Do NOT write prose; write a directive that the writer would expand into prose.>

[SUGGESTION 2]
LABEL: <...>
PROMPT: <...>

[SUGGESTION 3]
LABEL: <...>
PROMPT: <...>

Rules:

1. The three suggestions must be NARRATIVELY DISTINCT — different directions, not three rephrasings of the same idea. Variety matters more than quality.

2. Each PROMPT is a directive, not prose. Examples of correct directive voice:
   - "She runs into the alley behind the bar, terrified that he'll follow her. As she catches her breath, a stranger emerges from the shadows."
   - "Cut to morning. The night before is a blur. She wakes in an unfamiliar room and tries to piece together what happened."
   - "She decides to confront him directly. She walks back into the bar, locks eyes with him, and demands to know why he was watching her."

3. LABELs are pills. Keep them readable in 3-8 words. Examples: "She runs from the bar", "Cut to next morning", "She confronts him directly".

4. Do not use markdown, asterisks, or special formatting. Plain text only inside the LABEL and PROMPT fields.

5. Build naturally from where the story currently is. If the last assistant message ended with the protagonist arriving somewhere, the suggestions should be plausible next-actions from that point. Don't introduce out-of-context elements.

6. Output nothing outside the three [SUGGESTION N] blocks. No preamble. No commentary.`;
```

The `[SUGGESTION N]` block format is parser-friendly and matches the existing storyboard route's pattern (regex-parseable, robust to small LLM formatting drift). Don't ask for JSON — local LLMs are inconsistent at it.

### Part 5 — Suggestions API route

`src/app/api/chats/[id]/messages/[msgId]/suggestions/route.ts`:

POST endpoint, no body. Triggers a suggestion generation for the specified message.

Flow:

1. Fetch chat + messages.
2. Verify msgId exists, belongs to chat, role === 'assistant'.
3. Verify chat.suggestionsEnabled. If false, return 200 with `{ suggestions: null }` immediately (no LLM call).
4. Build the message context: walk active path from root to and INCLUDING the target assistant message. This is what the suggester sees.
5. Prepend the SUGGESTIONS_SYSTEM_PROMPT as the system message.
6. POST to Aphrodite (same WRITER_LLM_ENDPOINT as the main chat). Use lighter sampling — temperature 0.9, max_tokens 500, no DRY (different goal than creative prose; want clear structured output). Stream NOT required — get the full response in one shot.
7. Parse the response:

```ts
function parseSuggestions(text: string): Suggestion[] {
  const blocks = text.split(/\[SUGGESTION \d+\]/i).filter((b) => b.trim().length > 0);
  const suggestions: Suggestion[] = [];

  for (const block of blocks) {
    const labelMatch = block.match(/LABEL:\s*(.+?)(?:\n|$)/i);
    const promptMatch = block.match(/PROMPT:\s*([\s\S]+?)(?=\n\[SUGGESTION|$)/i);

    if (labelMatch && promptMatch) {
      const label = labelMatch[1].trim();
      const prompt = promptMatch[1].trim();
      if (label.length > 0 && prompt.length > 0) {
        suggestions.push({ label, prompt });
      }
    }

    if (suggestions.length === 3) break;
  }

  return suggestions;
}
```

8. If parsing yields fewer than 3 suggestions, return whatever was parsed (could be 0, 1, 2). The UI handles short lists gracefully.
9. Persist `Message.suggestionsJson` with the parsed array.
10. Return `{ suggestions: Suggestion[] }`.

Failure modes — return 200 with empty array, never throw to the client:

- LLM unreachable → `{ suggestions: [] }`
- LLM returns garbage → parser returns [], `{ suggestions: [] }` returned
- Timeout (10 seconds) → `{ suggestions: [] }`

The user never sees an error; pills just don't appear. Suggestions are an enhancement, not load-bearing.

Timeout: 10 seconds. Suggestions are a quick LLM task; if it takes longer, abandon. Use the existing AbortController + timeout pattern.

### Part 6 — Trigger the call from the client

In `ChatView.tsx`, find the streaming consumer's `done` handler. After the assistant message lands successfully, fire the suggestions request:

```ts
function consumeChatStream(/* ... */) {
  // ... existing handler ...

  case 'done': {
    if (assistantMsgId) {
      setMessages((prev) =>
        prev.map((m) =>
          m.id === assistantMsgId ? { ...m, content: finalContent } : m,
        ),
      );
    }
    setStreamingMsgId(null);
    setStreamingContent('');

    // NEW: kick off suggestions in the background
    if (chat.suggestionsEnabled && assistantMsgId) {
      void requestSuggestions(assistantMsgId);
    }
    break;
  }
}

async function requestSuggestions(messageId: string) {
  setSuggestionsLoading((prev) => new Set(prev).add(messageId));
  try {
    const res = await fetch(
      `/api/chats/${chatId}/messages/${messageId}/suggestions`,
      { method: 'POST' },
    );
    if (!res.ok) throw new Error('Suggestions request failed');
    const data = await res.json() as { suggestions: Suggestion[] };

    // Update local message state with suggestions
    setMessages((prev) =>
      prev.map((m) =>
        m.id === messageId ? { ...m, suggestionsJson: data.suggestions } : m,
      ),
    );
  } catch (err) {
    // Silent failure — pills just don't appear
    console.warn('Suggestions request failed:', err);
  } finally {
    setSuggestionsLoading((prev) => {
      const next = new Set(prev);
      next.delete(messageId);
      return next;
    });
  }
}
```

Same trigger from the regenerate and edit-and-continue flows — anywhere a new assistant message lands, suggestions kick off for it.

### Part 7 — Suggestions display: three pills above composer

`src/components/ChatView.tsx` — add a pills row above the composer.

Logic for which message's suggestions to display:
- The LAST assistant message on the active path (the most recent thing the LLM said)
- If the user is currently typing in the composer (textarea has any non-whitespace content), HIDE the pills
- If suggestions are still loading for that message, show skeleton pills
- If suggestions came back empty (parse failure or LLM returned nothing), no pills — just the composer

```tsx
const lastAssistantMsg = messages
  .filter((m) => m.role === 'assistant')
  .slice(-1)[0];

const showPills =
  chat.suggestionsEnabled &&
  lastAssistantMsg &&
  composerContent.trim().length === 0 &&
  !isRequestInFlight;

const pillsLoading = lastAssistantMsg && suggestionsLoading.has(lastAssistantMsg.id);
const pillsAvailable = lastAssistantMsg?.suggestionsJson ?? null;
```

Render:

```tsx
{showPills && (
  <div className="suggestions-row">
    {pillsLoading ? (
      <>
        <SuggestionSkeleton />
        <SuggestionSkeleton />
        <SuggestionSkeleton />
      </>
    ) : pillsAvailable && pillsAvailable.length > 0 ? (
      pillsAvailable.map((s, i) => (
        <SuggestionPill
          key={i}
          label={s.label}
          onTap={() => setComposerContent(s.prompt)}
        />
      ))
    ) : null}
  </div>
)}
```

### Part 8 — Tap behavior

When user taps a pill:

```ts
function handleSuggestionTap(suggestion: Suggestion) {
  setComposerContent(suggestion.prompt);
  // Optional: focus the composer so the user can immediately edit or send
  composerRef.current?.focus();
}
```

The pill's full prompt text replaces whatever's in the composer (which by definition was empty, since pills only show when composer is empty). User can:
- Send as-is (tap Send)
- Edit the prompt then send
- Clear and use a different pill
- Type their own thing instead

The pills DON'T disappear when one is tapped — they stay visible until the user starts typing OR sends a message. So if the user taps pill 1, then changes their mind and decides pill 3 is better, they can tap pill 3 and replace.

Once the user starts typing in the composer (any content beyond whitespace), pills hide. If they clear the composer to empty again, pills reappear.

### Part 9 — Branch switching invalidates pills

When the user switches branches via chevrons, the "last assistant message on active path" changes. The pills should now reflect THAT message's suggestions, not the previous active message's.

This happens automatically because:
- `messages` reflects all messages
- `resolveActivePath` returns a different path after branch switch
- `lastAssistantMsg` recomputes
- `pillsAvailable` reads from the new last message's `suggestionsJson`

If the new last message has cached suggestions: they appear immediately (no skeleton).

If it doesn't have cached suggestions (e.g., never been visited): silent — no pills appear. The user can manually request suggestions if a "regenerate suggestions" button exists, but I'd skip that for first cut. If they want suggestions, they continue the conversation and the next assistant message will get fresh ones.

Actually — for the case where a branch was ALREADY visited but suggestions failed: re-fire the suggestion request when this message becomes the active assistant tail. But cap at one retry per session to avoid hammering on a model that's not cooperating.

For first cut: skip retry logic. Pills only generate at message-creation time. If a message never got pills, it never has them.

### Part 10 — Don't fire on imported chats

When a chat is imported from SillyTavern (or backloaded somehow), all its assistant messages have `suggestionsJson === null`. The user doesn't expect 30 suggestion requests to fire on import.

The trigger only fires on `done` events from streaming flows. Imported messages don't go through streaming — they're inserted directly. So they never trigger suggestions.

If the user wants suggestions on the active tail of an imported chat: they continue the conversation, and the next response gets suggestions automatically. Acceptable.

## Out of scope

- A "regenerate suggestions" button to manually re-call the suggester. If suggestions are bad, the user types their own directive.
- Per-suggestion edit in the pill (e.g., long-press to expand-and-edit). Tap fills composer; user edits there.
- Diversity control over suggestions (e.g., "give me one safe, one weird, one funny"). Out of scope; can iterate the system prompt later if needed.
- Token cost display for suggestions. They're cheap; no need to surface.
- Streaming the suggestions endpoint. Two reasons not to: (a) suggestions are short and arrive fast, (b) UI shows skeleton during load. Not worth streaming.
- Suggestions for user messages (e.g., "here are three ways to ask the writer to expand"). The prompt is for next-directive suggestions only.
- Customizable suggestion count (1-5 instead of fixed 3). Three is the right number for cognitive load on tablet.
- A "shuffle" button for additional suggestions. If three aren't enough, the user types their own.
- Suggestions for the chat's initial directive (when no assistant message exists yet). The first directive is the user's; we don't suggest for empty conversations.
- Showing a hint label like "Tap a suggestion or type your own" above the pills. Pills are self-explanatory.
